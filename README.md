# Поиск статей 

Пайплайн retrieval + reranking для поиска релевантных статей помощи по тексту
пользовательского обращения. Используются готовые
предобученные модели (`intfloat/multilingual-e5-large` для dense-поиска,
`BAAI/bge-reranker-v2-m3` для реранкинга) поверх лексического поиска.

## 1. Обработка HTML статей

Статьи хранятся в поле `body` в виде HTML и требуют аккуратной очистки, чтобы
не потерять структуру (вкладки, спойлеры, таблицы), но и не оставить теги в
тексте:

1. HTML-entities разворачиваются (`html.unescape`), документ парсится через
   `BeautifulSoup` (`lxml`).
2. Служебные и нетекстовые теги (`script`, `style`, `nav`, `footer`, `svg`,
   `iframe`) вырезаются целиком.
3. `<br>` заменяются на перенос строки, а после каждого блочного тега
   (`p`, `div`, `li`, `table`, `td`, `h1`–`h6` и т.д.) добавляется `\n` —
   это сохраняет структуру статьи (таблицы, списки, спойлеры) в виде текста
   с переносами, а не «склеивает» всё в одну строку.
4. Убираются эмодзи молнии)(по unicode-диапазонам), неразрывные пробелы, лишние
   пробелы и повторяющиеся пустые строки.
5. Построчно убираются дубли подряд идущих одинаковых строк (частый артефакт
   вложенных `div`).

Результат — поле `body_clean`: чистый текст с сохранённой построчной
структурой, который используется дальше и для лексического, и для
dense-поиска.

## 2. Признаки, модели и алгоритмы

Пайплайн — гибридный: несколько независимых источников релевантности на
этапе retrieval, объединённые через Reciprocal Rank Fusion (RRF), и
дополнительный cross-encoder reranking поверх кандидатов.

### 2.1 Retrieval (кандидаты, top-100)

Пять независимых источников релевантности, каждый отдаёт top-100 кандидатов:

- `bm25_title` — BM25 по леммам заголовков статей;
- `bm25_body` — BM25 по леммам текста статьи целиком;
- `bm25_chunk` — BM25 по леммам чанков статьи, агрегация по статье через
  максимум;
- `char_tfidf` — TF-IDF по символьным 3–5-граммам (`title + body`);
- `dense` — эмбеддинги чанков (`multilingual-e5-large`), агрегация по
  статье через максимум.

Для лексического поиска текст лемматизируется (`pymorphy3`), приводится к
нижнему регистру, `ё`→`е`, чистится от стоп-слов и коротких токенов.

Для dense-поиска статьи режутся на **токен-ориентированные чанки** (бюджет
384 токена тела + 64 токена перекрытия, с учётом длины заголовка-префикса и
ограничения модели в 512 токенов), чтобы длинные статьи не обрезались и не
терялись целиком, а каждая часть текста участвовала в поиске. Эмбеддинги
считаются с префиксами `passage: ` / `query: `, как требует e5, и
кешируются на диск по хэшу конфигурации.

Все 5 списков (по 100 кандидатов) объединяются в единый `initial`-ранкинг
через **взвешенный RRF** (`rrf_k=60`), с весами источников (заголовок и
dense получают вес 1.35, остальные — 1.0–1.1), чтобы более точные сигналы
(заголовок, dense) весили больше без потери recall от остальных.

### 2.2 Reranking (top-100 → top-10)

Top-100 кандидатов из retrieval прогоняются через cross-encoder
`bge-reranker-v2-m3`:

- у каждой статьи-кандидата берётся не более 4 «лучших» чанков (по dense- и
  BM25-скору чанков поочерёдно), чтобы не гонять реранкер по всем чанкам
  длинных статей (до 447 чанков на статью) и уложиться во время;
- если пара запрос+чанк не влезает в лимит реранкера (1024 токена), чанк
  режется на перекрывающиеся куски (перекрытие 96 токенов) вместо обрезки —
  чтобы не терять концовку длинных ответов;
- по каждой статье берётся максимальный score среди её чанков/кусков —
  это финальный cross-encoder ранкинг (`cross`).

Финальный ответ — снова RRF-фьюжн `cross` и исходного `initial`-ранкинга,
но с большим весом у cross-encoder (`final_cross_weight = 3.0` против
`final_retrieval_weight = 1.0`), так как реранкер оказался существенно
точнее «сырого» retrieval (см. метрики ниже). RRF, а не прямое усреднение
score, выбран потому, что скоры разных источников (BM25, косинус, логит
реранкера) не сопоставимы по шкале.

## 3. Проверка качества на `calibration.f`

Все шаги (retrieval → reranking → фьюжн) прогоняются на размеченных 500
запросах из `calibration.f`, метрика — `MAP@10` (плюс `Recall@10/100` как
диагностика).

Результаты на калибровке:

- Retrieval (RRF из 5 источников): Recall@100 = 0.986, Recall@10 = 0.704,
  MAP@10 = 0.351;
- Cross-encoder (только реранк, без фьюжна с retrieval): MAP@10 = 0.524;
- Финальный fused ранкинг (cross + retrieval через RRF): Recall@10 = 0.886,
  **MAP@10 = 0.537**.

Дополнительно строится построчный отчёт `calibration_report.csv`
(`query_id`, `query_text`, `ground_truth`, `prediction`, `ap_at_10`) —
позволяет вручную просматривать запросы с низким AP@10 и разбирать
конкретные промахи, а не только смотреть на усреднённую метрику.

## 4. Найденные типы ошибок и что было сделано

Анализ retrieval- и reranking-этапов по отдельности (`Recall@100 = 0.986`
против `MAP@10 = 0.351` на чистом retrieval) показал, что основная проблема —
не в том, что нужная статья не находится, а в том, что она плохо
ранжируется среди кандидатов. Отсюда и конкретные ошибки:

- **Слабое ранжирование лексическим/dense-поиском по отдельности.**
  BM25 и dense-поиск давали высокий recall (нужная статья почти всегда
  попадала в топ-100), но частo не в топ-10. Решение — добавить
  cross-encoder реранкинг поверх кандидатов, что подняло MAP@10 с 0.351 до
  0.524 на одном только cross-скоре.
- **Длинные статьи «размывают» сигнал.** У части статей до ~450 чанков,
  из-за чего при усреднении/суммировании по чанкам релевантный фрагмент
  тонет среди нерелевантных. Решение — агрегация по чанкам статьи через
  **максимум**, а не среднее (и для BM25-чанков, и для dense), плюс отбор
  только top-4 «лучших» чанков на статью перед реранкингом.
- **Разная роль заголовка и тела статьи.** Заголовок оказался более сильным
  сигналом релевантности для коротких обращений, чем весь текст статьи —
  поэтому `bm25_title` и `dense` получили повышенный вес (1.35) в RRF, а не
  равный вес со всеми источниками.
- **Длинные ответы/чанки не влезают в лимит реранкера.** Обрезка привела бы
  к потере концовки длинных инструкций. Решение — разбиение пары
  запрос+чанк на перекрывающиеся куски (96 токенов перекрытия) вместо
  truncate, с выбором максимального скора по кускам.
- **Несопоставимость шкал скоров разных источников** (BM25-скор, косинусное
  сходство, логит cross-encoder) — прямое усреднение/суммирование скоров
  было бы некорректным. Решение — везде использован **RRF** (по рангу, а не
  по значению скора), что даёт устойчивое объединение разнородных ранкингов.
- **Разговорный, неформальный текст обращений** (опечатки, обращения вроде
  «Добрый день», сленг) слабее ловится точным лексическим совпадением.
  Решение — лемматизация + стоп-слова для BM25 и добавление
  `multilingual-e5-large` как семантического источника, устойчивого к
  перефразированию.

Итоговый пайплайн и построчный отчёт `calibration_report.csv` оставлены как
инструмент для дальнейшего точечного разбора отдельных промахов на
`test.f`.
